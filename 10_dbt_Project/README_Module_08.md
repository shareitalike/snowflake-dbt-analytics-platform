# Phase 10 - Module 8: Enterprise Snapshot Framework

This module establishes the SCD Type 2 (Slowly Changing Dimensions) engine of our Enterprise Data Platform, utilizing native dbt Snapshots to preserve the historical truth of our data.

## Deliverables Checklist

- [x] **Repository Structure:** Models placed in `snapshots/customer/` and `snapshots/product/`.
- [x] **Timestamp Strategy (`snap_customer`):** Configured to track mutations using the `dbt_updated_at` timestamp. This is the highest-performing strategy in Snowflake.
- [x] **Check Strategy (`snap_product`):** Configured to track mutations explicitly against the `is_product_active` and `warehouse_id` columns. This prevents "Snapshot Bloat" by ignoring superficial `updated_at` bumps generated by upstream ERP batch jobs.
- [x] **Defensive Configurations:** Implemented `invalidate_hard_deletes=True` globally to ensure deleted source records are properly expired in the warehouse, maintaining compliance with PII/GDPR mandates.
- [x] **Operational Metadata:** Maintained compliance with the `AGENTS.md` rules, embedding Expected Row Counts and Refresh Frequencies directly into the snapshot configurations and `schema.yml`.
- [x] **Architecture Documentation:** Authored the [Design Summary](file:///F:/snowflake/Project_snow_live/Project_snowflake_live/10_dbt_Project/docs/Module_08_Design_Summary.md), [Operational Runbook](file:///F:/snowflake/Project_snow_live/Project_snowflake_live/10_dbt_Project/docs/Module_08_Runbook.md).md) detailing how to manage Snapshot Bloat and how Surrogate Keys link historical Fact data to the correct historical Dimension state.

## Usage Example (Executing Snapshots)
```bash
# Run all snapshots
dbt snapshot --target dev
```
