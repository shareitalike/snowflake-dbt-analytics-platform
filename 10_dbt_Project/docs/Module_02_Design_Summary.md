# Enterprise Sources & Seeds Framework
## Module 02 - Design Summary

### Why Sources?
In dbt, `Sources` are the abstraction layer over the raw data sitting in the Snowflake `Bronze` databases. Hardcoding `select * from DB_PROD_RAW.SC_SHOPIFY.TB_ORDERS` inside SQL models creates fragile, untestable code. By declaring sources in YAML:
1. We unlock **Source Freshness** checking (e.g. "Alert me if the Stripe API hasn't dumped data in 6 hours").
2. We unlock **Source Testing** (e.g. "Fail the dbt build if Shopify Order IDs are null or duplicated").
3. We establish **Source Contracts** that explicitly document the incoming schema expected from the Data Engineering team.

### Why Seeds & Reference Data Strategy?
`Seeds` are CSV files version-controlled directly within the dbt git repository. They are used exclusively for **static reference data** (e.g., Country Codes, fixed Tax Rates).
- **Why?** It democratizes reference data. Instead of requiring a DBA to run `INSERT` scripts in production to add a new Promotion Type, an Analytics Engineer simply adds a row to `promotion_types.csv` and merges a PR. dbt handles the physical `CREATE/INSERT` on the next `dbt seed` run.
- **Limitation:** Seeds should *never* be used for large dimensions or fast-changing data. They are strictly for tiny mapping tables (typically under 1,000 rows).

### Source Freshness Strategy
For mission-critical retail domains (Orders, Inventory, Payments), freshness is a tier-1 SLA. Our source definitions use `loaded_at_field` to ping the `metadata_inserted_at` timestamp generated by the Phase 8 CDC framework. We configure distinct `warn_after` (Slack alert) and `error_after` (PagerDuty alert) thresholds.
