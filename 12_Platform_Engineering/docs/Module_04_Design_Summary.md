# Platform Engineering - Secrets & Configuration Management
## Module 04 - Design Summary

### Zero Trust Principles
In an enterprise data platform, hardcoded credentials are a massive vulnerability. We strictly enforce a Zero Trust architecture by completely separating code from configuration. 
No Python script, dbt macro, or Airflow DAG contains a password or API token. 

### AWS Secrets Manager & KMS Encryption
We utilize AWS Secrets Manager to store the credentials for Snowflake, dbt Cloud, Slack, and Power BI. To ensure data at rest is secure, these secrets are not encrypted with standard AWS keys; they are encrypted using a dedicated Customer Managed Key (CMK) generated by our `aws/kms` Terraform module. This allows us to strictly audit exactly who (or what IAM Role) encrypts and decrypts our platform secrets.

### GitHub OIDC Integration (Zero-Touch Cloud Credentials)
Historically, to allow GitHub Actions to deploy to AWS, engineers had to generate a static AWS Access Key and store it in GitHub Secrets. This led to keys sitting unrotated for years.
We replaced this with **OpenID Connect (OIDC)**. GitHub Actions requests a short-lived JSON Web Token (JWT) directly from AWS IAM. AWS validates the token's origin (e.g. `repo:omniretail/enterprise-data-platform:main`), and if it matches the trust policy, it grants temporary access. No long-lived keys exist.

### Airflow Integration
Airflow is configured to use the `SecretsManagerBackend`. When the Master DAG attempts to trigger a Snowpark job, it asks Airflow for the `snowflake_default` connection. Airflow reaches out to AWS Secrets Manager at runtime, fetches the Snowflake credentials, authenticates, and purges the secret from memory.
